Post-ChatGPT, we were working in about 4,000 tokens. That's it. Barely enough to do anything substantial.

THE ORIGIN STORY◆ 04
Step one upContext era

Answer #1: context engineering

Recycle that tiny window smartly. Three moves powered the first wave of coding agents:

🛠️

Tool calling

Explore the repo, read only the relevant files, and take actions in the outside world.

🔌

MCP

Bolt vendor-specific features on top of the base model.

📚

RAG

Connect custom databases for on-demand data, available any time.

This birthed the early players: Cursor, Windsurf, Cline, Roo, Aider. Genuinely good tools that got the job done.

CONTEXT ENGINEERING◆ 05
Then it brokeThe wall

Bigger context → bigger asks → things fall apart

Windows grew, so people asked for huge tasks: "clone this entire website."

  • Sites came back half-finished.
  • Buttons that just don't work.
  • Features never tested all the way through.

⚠️ THE SUMMARIZATION TRAP

As the window fills on a 12-hour task, the agent summarizes its own context to keep going.

So it's bound by how well it summarizes — and often assumes a task is done when it never was.

WHY CONTEXT ENG WASN'T ENOUGH◆ 06
Everyone reached the same ideaConverge

The fixes all pointed the same direction

🪆

Sub-agents

Hierarchical context — a parent delegates to children with their own windows.

🐝

Agent swarms

Many agents, each with its own context, working in parallel.

Add it up — orchestration + a real execution environment + context management — and you're already harnessing the agent.

The pattern existed for years. The term "agent harness" just got coined in early 2026.

CONVERGING ON THE HARNESS◆ 07
The one big ideaLoops

The shift = the loop

Step one layer above context engineering. Run the agent in a loop where every iteration gets a fresh, clean context — under strict rules for how to start and finish.

No more slowly-rotting summary. Each pass starts clean.

🔁
RALPH

Took over the internet — not because it was clever, but because the architecture was dead simple.

THE PARADIGM CHANGE◆ 08
Important nuanceIt's a stack

Harness doesn't replace the others — it sits on top

🗣️ Prompt engineeringStill here. Gives the agent its persona — "you are a coding agent." Smaller piece now.
🧠 Context engineeringStill here. Loads the right files, tools, and data into each window.
🎛️ Harness engineeringNew layer. The looped environment that drives the whole thing — and leverages both below it.

Open up Cline and the system prompt is still a well-written prompt. Nothing got deprecated — it got wrapped.

THE LAYER CAKE◆ 09
What it looks like in practiceThe pattern

The canonical harness loop

  • Generate a big requirements file (a PRD).
  • Loop: pick exactly one task from it.
  • Implement it, test it, document it.
  • Fresh prompt + fresh context, repeat till done.

Ralph & Anthropic's harness demo do exactly this — and both repos are tiny. Simple is the point.

# the whole architecture, basically make requirements.json PRD → tasks while tasks_remain: task = pick_one() just one! implement(task) test(task) verify document(task) reset_context() fresh start → done, for real this time

WHAT IT IS◆ 12
The load-bearing decision2 planes

One screen, fed by two data planes

⌨️

Terminal plane

Real node-pty processes. Byte-for-byte authentic claude output, streamed over IPC and rendered in xterm.js. You can type back in.

📡

Event plane

Claude Code hooks (PreToolUse, PostToolUse, Stop…) POST to a Unix socket → drive the avatars and the hive.

Hooks alone can't show the raw stream. The stream alone can't tell you which tool is running. Together = the full picture.

ARCHITECTURE◆ 13
The harness layerThe Hive

The Hive = the harness around a team

  • Memory — each agent reads/appends its own memory.md; instant semantic recall.
  • Mailboxes — atomic one-file-per-message inbox/outbox (actor model).
  • Blackboard — a shared plan agents co-author.
  • Router — moves messages between agents; single git committer so nothing corrupts.

CONCEPT → REALITY◆ 16
Where it gets realGuardrails

The unglamorous code that keeps 15 agents from imploding

🔒 Single committer

Only the main process touches git → no index.lock corruption with many agents.

🛑 Stop-hook loop

A finished agent drains its inbox and keeps working — guarded so it can't loop forever.

🏓 Hop caps

Only request/query/propose obligate a reply; past a cap, GOD escalates instead of ping-pong.

📥 Idempotent mail

Per-agent cursor.json; re-seeing a message id is a no-op.

🧯 Degrade gracefully

No semantic index installed? Markdown memory still works. Nothing hard-fails.

🙋 HITL escalation

Spend / destructive / scope changes land in an approvals queue — a human clears them.
